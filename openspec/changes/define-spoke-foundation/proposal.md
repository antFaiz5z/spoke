# 提案：确定 Spoke 基础方案

## 概述

为 `spoke` 确定核心产品定义与技术基础。`spoke` 被定义为一个基于场景分类、以结构化文本片段为中心的英语口语练习产品，而不是一个以聊天消息为核心的应用。

本次变更在实现前先固定 V1 方向：

- `spoke` 是一个以文本片段为中心的练习产品
- 前端采用 `Next.js + React`
- 样式方案优先采用 `Tailwind CSS`
- 后端优先采用 `TypeScript`，V1 使用 `Next.js Route Handlers`
- 练习内容以只读的结构化文本树渲染
- 预置内容和用户生成内容都存入数据库
- 主应用数据库采用 `PostgreSQL`

## 背景

产品主交互链路是：

- 浏览场景化练习内容
- 打开一个练习单元
- 命中段 / 句 / 词文本片段
- 悬浮、高亮、点击，并通过 TTS 播放片段

这种交互更像“文档播放器 + 练习界面”，而不是聊天应用。因此需要尽早锁定产品边界，避免后续实现逐渐偏向 chat-first 架构。

## 范围

本次变更定义：

- 产品定位
- V1 前端框架选型
- V1 内容模型
- V1 数据库策略
- V1 API 形状与前端消费原则

本次变更不定义：

- 视觉设计
- 认证体系
- 练习历史详情
- 发音评分
- 自定义后台
- 细化的 TTS 缓存策略

## 已确定决策

### 产品核心

`spoke` 是一个围绕结构化文本内容构建的场景化英语口语练习产品，核心对象为：

- 场景
- 练习内容
- 段落
- 句子
- 词元

它不被定义为聊天优先产品。

### 前端

主前端采用：

- `Next.js`
- `React`
- `Tailwind CSS`

### 后端

V1 后端优先采用：

- `TypeScript`
- `Next.js Route Handlers`

### 渲染模型

练习界面采用：

- 只读渲染
- 自行管理段 / 句 / 词 span 结构
- 基于距离驱动的 `paragraph | sentence | token` 层级命中方式
- 单主命中节点与独立播放状态

富文本编辑器框架不在 V1 范围内。

### 内容存储

预置材料和用户生成材料都存入数据库。

### 数据库

主数据库采用 `PostgreSQL`。

V1 中：

- `Scenario` 与 `ContentItem` 采用关系化存储
- 结构化文本树存放在 `ContentItem.structured_content` 的 `jsonb` 字段中

## 预期结果

在本次变更之后，后续实现可以基于稳定基础推进，而不必反复重议以下问题：

- 产品究竟是 chat-first 还是 text-first
- 主数据库用 SQLite 还是 PostgreSQL
- V1 是否要把段 / 句 / 词全部规范化成独立数据表
